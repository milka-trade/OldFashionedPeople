## 🏗️ 구현된 3-Tier 정책

### 정책 1: `state_edge` (지속 상태)
```
상태 전이만 알림 + 장기 지속 시 에스컬레이션 + 회복 시 Recovery
```
**적용**: 현금 부족, (미래 추가 가능: WebSocket 끊김, API 장애 등)

### 정책 2: `event_dedup` (반복 사건)
```
같은 키의 사건은 dedup_window_sec 동안 침묵 + 억제 횟수 표시
```
**적용**: 시스템 에러 (5분), 수동매도 추정 (60분)

### 정책 3: `always` (사건성)
```
무조건 발송 + Discord 2초당 5요청 한도만 준수
```
**적용**: 매수/매도 체결, 봇 시작/종료, 정시 보고

## 📚 학술적 근거 / 출처

### 핵심 SRE 원칙

1. **Alert Fatigue (알림 피로)** — Google SRE Workbook  
   "부적절한 알림과 저가치 알림은 가장 시급한 문제 중 하나. 현대 옵저버빌리티는 막대한 텔레메트리를 생성하므로 지능적인 필터링 없이는 매일 수천 개의 보고서가 쏟아진다"

2. **State vs Event 구분** — Prometheus Alertmanager 설계 철학  
   "Alertmanager는 알림 피로 문제를 해결하기 위해 만들어졌다. 단순 라우팅 도구가 아니라 알림을 관리, 통합, 지능적으로 억제하기 위한 정교한 시스템"

3. **Hysteresis (히스테리시스)** — incident.io 2025 모범 사례  
   "히스테리시스와 시간 기반 억제를 구현하면 노이즈를 완화한다"

4. **Deduplication (중복 제거)** — Squadcast SRE 가이드  
   "중복 제거는 동일한 알림을 단일 알림으로 결합하는 것. 60분 동안 매분 실패한 체크에 대해 60개 이메일이 아닌 'Check failed (x60)' 알림 1개를 받아야 한다"

### Discord 플랫폼 제약

5. **Webhook Rate Limit** — Discord 공식 문서  
   "웹훅당 2초당 5요청. 실패 요청도 한도에 포함"

6. **Cloudflare IP Ban Risk** — Hookdeck Best Practices  
   "단일 IP에서 10분 내 10,000요청이 401, 403, 429 상태 코드를 반환하면 Cloudflare가 IP 전체를 차단"

### 트레이딩 봇 구체적 적용

7. **Throttling Pattern** — Discord Developer Support  
   "조절(Throttling)은 한도에 도달하기 전에 요청 속도를 제어하는 사전 예방적 접근법"

8. **3-Tier Severity** — incident.io 2025  
   "3-Tier 에스컬레이션 모델은 알림 심각도를 적절한 응답에 맞춘다"

### 알림 노이즈 감소 검증 지표

9. **Healthy Actionable Rate** — incident.io  
   "실행 가능한 알림이 10% 미만이면 노이즈가 심각. 건강한 시스템은 30~50% 실행 가능 비율을 달성"

   → v36 현재 상태: 1일 144회 노이즈 / 5~10회 실행 가능 = **3% 실행 가능률 (위험)**  
   → 패치 후: 1일 5~10회 노이즈 / 5~10회 실행 가능 = **50%+ 실행 가능률 (건강)**
